Goal
----
Push local ARI well above the 0.63 Kaggle baseline (currently at 0.62 from
zero-shot Cellpose cpsam in pipeline.py). Phase 1 deliverable: a local
validator that scores any segmentation callable against a committed set of
held-out training FOVs using the *official* metric.

Key Data-Structure Findings (before writing any code)
-----------------------------------------------------
1. spots_train.csv has columns:
     barcode_id, fov, image_row, image_col, global_x, global_y, global_z,
     x, y, target_gene
   There is NO cell_id / cluster_id column. The user's spec anticipated one
   but it does not exist. Ground-truth spot->cell assignment must be derived
   via point-in-polygon against cell_boundaries_train.csv.

2. cell_boundaries_train.csv has 4082 rows, one per cell, with polygon
   coordinates as comma-separated strings in columns
     boundaryX_z{0..4}, boundaryY_z{0..4}
   Each cell_id (column 'Unnamed: 0', long numeric string) matches exactly the
   obs/_index in counts_train.h5ad, which is where we pull the cell->FOV map
   from (obs/fov).

3. Pixel conversion, verified empirically on 10 spots from FOV_019 against the
   precomputed image_row / image_col columns:
     image_row = 2048 - (global_x - fov_x) / pixel_size
     image_col =         (global_y - fov_y) / pixel_size
   The stage x-axis corresponds to the INVERTED image row, not the column —
   a common source of silent bugs. Per-FOV (fov_x, fov_y, pixel_size) come
   from reference/fov_metadata.csv. pixel_size = 0.109 µm/px everywhere.

4. global_z takes exactly 5 integer values {0, 1, 2, 3, 4}. Spot counts:
   z0: 589K, z1: 606K, z2: 580K, z3: 521K, z4: 362K. So GT rasterization has
   to be done per-z, not just at the middle plane the current pipeline uses.

5. Official metric (per /scratch/pl2820/data/competition/metric.py):
   - Input: solution.csv with [spot_id (index), fov, gt_cluster_id]
            submission.csv with [spot_id (index), fov, cluster_id]
   - For each fov in solution: adjusted_rand_score(gt, pred) on string labels.
   - Mean across FOVs. ARI is cluster-ID independent, so the 'cluster_N' vs
     'FOV_X_cell_N' naming choice doesn't matter numerically.
   - Missing spots in submission are filled with 'background'.
   - Organizer-provided generate_submission.py uses 'FOV_X_cell_N' format;
     we follow the same format for compatibility.

Deliverables — Files Created
----------------------------
scripts/build_gt_labels.py
  One-off GT builder. Rasterizes cell_boundaries_train.csv into a (2048, 2048)
  int32 mask per (FOV, z) with cv2.fillPoly, then looks up every training
  spot at its (image_row, image_col, z_idx) to assign a gt_cluster_id.
  Output: cache/gt_spot_labels.parquet (2,659,359 rows, ~13 MB, 17 s to build).
  Schema: spot_idx (int64), fov (str), gt_cluster_id (str).
  Background labels use the literal string 'background'; cell labels use
  '{fov}_cell_{original_cell_id}'.
  Overall: 72.8% of all training spots are background (outside any polygon).

scripts/make_val_split.py
  Deterministic stratified picker. Sorts the 40 training FOVs by cells-per-FOV
  (ascending; tie-break by spots-per-FOV) and picks 6 at evenly-spaced
  quantile positions (linspace 0..N-1). Writes val_fovs.txt.
  Fixed committed split:
     FOV    cells  spots    GT bg%
     FOV_012   53  78,605   80.3
     FOV_013   72  54,197   69.8
     FOV_031   83  87,881   77.7
     FOV_015   95  52,204   72.4
     FOV_032  114  55,973   71.9
     FOV_019  246  72,619   73.1
  Spans 53 -> 246 cells (full density range). 401,479 total val spots.

scripts/segmenters.py
  Factory registry used by local_eval.py. Each factory takes (gpu, diameter)
  and returns a closure segment(fov_dir: Path) -> (2048, 2048) int mask. The
  model lives in the closure so it's built once and reused across FOVs.
  Available factories:
    build_all_background   — no-op, returns all-zeros. Plumbing smoke test.
    build_cellpose_zeroshot — wraps pipeline.py's segment_fov with cpsam.

scripts/local_eval.py
  Main validator. Arguments:
    --segmenter module.path:factory_name  (required)
    --val_fovs path  (default val_fovs.txt)
    --data_root path  (default /scratch/pl2820/data/competition)
    --gt_labels path  (default cache/gt_spot_labels.parquet)
    --gpu  (forwarded to the factory)
    --diameter float  (negative = auto; forwarded to the factory)
    --z_filter int  (NEW — diagnostic: restrict val spots to one z-plane
                     only. Filters both spots and GT in lockstep so the
                     length-match assertion still holds. Intended to
                     isolate per-z segmentation lift from over-/under-
                     segmentation errors.)
    --save_masks  (dumps predicted masks for debugging)
    --run_name name  (override runs/<name> dir)
  Workflow:
    1. Parse factory spec, build segmenter.
    2. Load cached GT for val FOVs only.
    3. For each val FOV: run segmenter, lookup spots in mask, assign
       '{fov}_cell_N' or 'background'.
    4. Log per-FOV: n_spots, n_pred_cells, n_gt_cells, pred_bg%, gt_bg%,
       ARI, seg_seconds.
    5. Build solution + submission DataFrames and call the organizer's
       metric.merfish_score VERBATIM (imported from
       /scratch/pl2820/data/competition/metric.py). This is the headline
       number. A parallel sklearn-based per-FOV ARI is logged for
       diagnostics; if the two disagree by > 1e-6, a warning is emitted.
  Run outputs in runs/<timestamp_or_name>/:
    config.json  — args + cuda_available + start time
    eval.log     — full log
    per_fov.csv  — one row per FOV with the stats above
    summary.json — mean_ari (official), mean_ari_inline, per_fov_ari
    masks/<fov>.npy  — only if --save_masks

run_local_eval.sh
  SLURM launcher: gpu:h100:1, 4 cpus, 32 GB, 2 h wall. Runs inside the
  existing singularity image and my_writable_env conda env. Passes --gpu by
  default; forwards any additional arguments to local_eval.py.
  Overlay is mounted :ro so multiple jobs can share it concurrently — needed
  for fan-out runs like the Phase 2 diameter sweep. cellpose is assumed to
  already be installed in my_writable_env (the pip install step was removed
  with the :ro switch).
  Invocation: sbatch run_local_eval.sh --run_name phase1_baseline_zeroshot

scripts/compute_diameter.py                                        (Phase 2)
  Calibrates Cellpose's `diameter` hint from GT polygons. Parses
  cell_boundaries_train.csv, computes per-cell equivalent-circle diameter
  d = 2*sqrt(area_px/pi) via the shoelace formula on the stage-µm polygon
  (area_px = area_µm² / pixel_size² — the affine transform is isotropic so
  no per-FOV offsets are needed). Defaults to z=2 (the plane pipeline.py
  currently segments at); --all_z prints per-z stats; --z overrides. Writes
  the median to reference/diameter_px.txt.
  Result (z=2, 3774 / 4082 cells with valid polygon, 308 missing):
    mean=86.38 median=87.68 p25=74.76 p75=98.70 p90=107.79 (all in pixels)
  Cross-z distributions are tight (median per z: 90.6 / 87.4 / 87.7 / 89.6 /
  92.7), so z=2 is a fair single estimate.

scripts/prep_training_data.py                                      (Phase 3)
  Materializes Cellpose fine-tuning inputs. For each training FOV (40 - 6
  val = 34 FOVs) loads DAPI frame 16 + polyT frame 15, normalizes via
  pipeline.normalize_image, and rasterizes the z=2 polygon through
  build_gt_labels.rasterize_fov_z (exact reuse — training masks are
  bit-identical to what local_eval.py scores against). Writes
  training_data/<fov_id>.npz with img (H, W, 3) float32 [polyT, DAPI, zeros]
  and mask (H, W) int32 (0 = bg, 1..N cells). Also renders 4 stratified QC
  PNGs under training_data/qc/<fov>.png showing DAPI + z=2 polygon outlines.
  Run result (40.6 s CPU): 34 FOVs prepared, 0 failures, 3419 GT cells ->
  3177 rasterized at z=2 (7.1% dropout for cells whose z=2 polygon is
  degenerate/empty). QC overlay on FOV_029 (dense tissue) confirms polygons
  land on DAPI nuclei throughout.

scripts/segmenters.py  (+ build_cellpose_finetuned)                (Phase 4)
  Added a second factory, build_cellpose_finetuned(gpu, diameter,
  pretrained_model, **_), that wraps Cellpose on a custom weights path.
  Required `pretrained_model` is supplied via local_eval.py's new
  --segmenter_kwargs flag. Existing factories now accept **_ so they
  silently ignore unknown kwargs forwarded by local_eval.py.

scripts/local_eval.py  (+ --segmenter_kwargs)                      (Phase 4)
  New flag: --segmenter_kwargs "key=value,key=value" — parsed into a dict
  and forwarded to the factory alongside gpu/diameter. Values are kept as
  strings; factories coerce as needed. Recorded in config.json and
  summary.json for provenance. Example:
    --segmenter_kwargs "pretrained_model=/scratch/.../runs/phase4_v1/best.pt"

scripts/train_cellpose.py                                          (Phase 4)
  Fine-tunes cpsam on the Phase 3 .npz files. Deterministic Phase-4 val
  pick: sorts the 34 training FOVs by cell count and takes the inner-1/3
  and inner-2/3 positions as the val set (count=2 by default), leaving 32
  for training. The Phase 1 val FOVs in val_fovs.txt are NOT used here —
  they are reserved for final ARI eval on this run's best weights.
  Calls cellpose.train.train_seg with the requested hyperparameters
  (n_epochs=100, lr=1e-5, weight_decay=0.1, batch_size=8, normalize=False,
  channel_axis=-1, save_every=10, save_each=True). After training, the
  intermediate checkpoint whose eval epoch has the lowest test_loss is
  promoted to runs/<name>/best.pt; the final-epoch checkpoint is also
  always kept at runs/<name>/models/<model_name>. SGD flag is exposed for
  parity with the Phase 4 spec but cellpose v4.0.1+ always uses AdamW.
  Outputs per run (runs/<name>/):
    config.json        — all hyperparams + val_fovs + started_at
    train.log          — full training log (cellpose train_logger + this script)
    losses.csv         — (epoch, train_loss, test_loss) per epoch
    models/<name>      — final-epoch checkpoint
    models/<name>_epoch_NNNN — intermediate checkpoints
    best.pt            — copy of the best-by-test_loss intermediate
    summary.json       — best_epoch, best_loss, final losses, val_fovs

run_train_cellpose.sh                                              (Phase 4)
  SLURM launcher for Phase 4: gpu:h100:1, 8 cpus, 64 GB, 6 h wall. Overlay
  mounted :ro. Runs training then, on success, invokes local_eval.py with
  build_cellpose_finetuned pointed at runs/<name>/best.pt plus the
  Phase-2-calibrated diameter (default 87.68 px; override via EVAL_DIAMETER
  env var). RUN_NAME env var controls the run dir name (default
  phase4_<timestamp>). Any flags after the script name are forwarded to
  train_cellpose.py. Example:
    RUN_NAME=phase4_v1 sbatch run_train_cellpose.sh
    RUN_NAME=phase4_v1_lr5e-6 EVAL_DIAMETER=96.45 \
        sbatch run_train_cellpose.sh --learning_rate 5e-6

Verification Performed
----------------------
- Cell ID sets between counts_train.h5ad obs/_index and cell_boundaries_train
  match exactly (4082 in both, intersection 4082, no orphans).
- Pixel-conversion formula cross-checked on 10 distinct spots in FOV_019;
  predicted image_row/image_col match the precomputed values to sub-pixel
  precision (rounding only).
- GT build succeeds on all 40 FOVs, 2.66 M spots, in 17.2 s total CPU.
- Smoke test with build_all_background (all-zeros mask) on the 6 val FOVs:
    mean_ari_inline = 0.0000
    mean_ari (official metric.merfish_score) = 0.0000
  Inline and official computations agree bit-exact, confirming the GT join,
  per-FOV ARI reduction, and submission-shape construction are correct.

- Phase 1 baseline GPU run (runs/phase1_baseline_zeroshot/) — zero-shot
  cpsam, auto diameter, 6 val FOVs:
    mean_ari (official) = 0.4765
    per-FOV: FOV_012=0.329  FOV_013=0.509  FOV_031=0.384
             FOV_015=0.705  FOV_032=0.645  FOV_019=0.286
  Substantially below the 0.62 Kaggle leaderboard number. The worst FOVs are
  the density extremes (FOV_019 densest at 246 cells, FOV_012 sparsest at
  53). Per the summary's own rule, this says the 6-FOV split is
  unrepresentative of the test distribution, not that cpsam is broken —
  future comparisons should treat the *delta* on this split as the signal
  and trust the Kaggle LB as the anchor on absolute level.

Outstanding

- Z-plane diagnostic (interleaved with Phase 2). Does restricting the val
  spots to z=2 only — the plane cpsam actually sees — materially move ARI?
  If yes, per-z segmentation is a bigger lever than currently estimated and
  the roadmap should be reordered in favor of it. If no, over-/under-
  segmentation is the dominant error and diameter/fine-tune remain the
  right next steps. Runs as:
      sbatch run_local_eval.sh --run_name phase1_zcheck_z2only --z_filter 2
  Caveat: pipeline.py loads DAPI_FRAME=16 / POLY_T_FRAME=15 from the .dax,
  which the project assumes is z=2. Worth verifying the frame->z map before
  over-interpreting the delta.

- Phase 2 diameter sweep. Submit in parallel (overlay is :ro now):
      sbatch run_local_eval.sh --run_name phase2_diameter_median --diameter 87.68
      sbatch run_local_eval.sh --run_name phase2_diameter_0.9x   --diameter 78.91

      sbatch run_local_eval.sh --run_name phase2_diameter_1.2x   --diameter 105.22

  0.4765. Deliverable: report best diameter + per-FOV breakdown.

  pressing further — the 0.14 gap vs Kaggle LB is large enough that the
