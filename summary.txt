Key Data-Structure Findings (before writing any code)
-----------------------------------------------------
1. spots_train.csv has columns:
     barcode_id, fov, image_row, image_col, global_x, global_y, global_z,
     x, y, target_gene
   There is NO cell_id / cluster_id column. The user's spec anticipated one
   but it does not exist. Ground-truth spot->cell assignment must be derived
   via point-in-polygon against cell_boundaries_train.csv.

2. cell_boundaries_train.csv has 4082 rows, one per cell, with polygon
   coordinates as comma-separated strings in columns
     boundaryX_z{0..4}, boundaryY_z{0..4}
   Each cell_id (column 'Unnamed: 0', long numeric string) matches exactly the
   obs/_index in counts_train.h5ad, which is where we pull the cell->FOV map
   from (obs/fov).

3. Pixel conversion, verified empirically on 10 spots from FOV_019 against the
   precomputed image_row / image_col columns:
     image_row = 2048 - (global_x - fov_x) / pixel_size
     image_col =         (global_y - fov_y) / pixel_size
   The stage x-axis corresponds to the INVERTED image row, not the column —
   a common source of silent bugs. Per-FOV (fov_x, fov_y, pixel_size) come
   from reference/fov_metadata.csv. pixel_size = 0.109 µm/px everywhere.

4. global_z takes exactly 5 integer values {0, 1, 2, 3, 4}. Spot counts:
   z0: 589K, z1: 606K, z2: 580K, z3: 521K, z4: 362K. So GT rasterization has
   to be done per-z, not just at the middle plane the current pipeline uses.

5. Official metric (per /scratch/pl2820/data/competition/metric.py):
   - Input: solution.csv with [spot_id (index), fov, gt_cluster_id]
            submission.csv with [spot_id (index), fov, cluster_id]
   - For each fov in solution: adjusted_rand_score(gt, pred) on string labels.
   - Mean across FOVs. ARI is cluster-ID independent, so the 'cluster_N' vs
     'FOV_X_cell_N' naming choice doesn't matter numerically.
   - Missing spots in submission are filled with 'background'.
   - Organizer-provided generate_submission.py uses 'FOV_X_cell_N' format;
     we follow the same format for compatibility.

Deliverables — Files Created
----------------------------
scripts/build_gt_labels.py
  One-off GT builder. Rasterizes cell_boundaries_train.csv into a (2048, 2048)
  int32 mask per (FOV, z) with cv2.fillPoly, then looks up every training
  spot at its (image_row, image_col, z_idx) to assign a gt_cluster_id.
  Output: cache/gt_spot_labels.parquet (2,659,359 rows, ~13 MB, 17 s to build).
  Schema: spot_idx (int64), fov (str), gt_cluster_id (str).
  Background labels use the literal string 'background'; cell labels use
  '{fov}_cell_{original_cell_id}'.
  Overall: 72.8% of all training spots are background (outside any polygon).

scripts/make_val_split.py
  Deterministic stratified picker. Sorts the 40 training FOVs by cells-per-FOV
  (ascending; tie-break by spots-per-FOV) and picks 6 at evenly-spaced
  quantile positions (linspace 0..N-1). Writes val_fovs.txt.
  Fixed committed split:
     FOV    cells  spots    GT bg%
     FOV_012   53  78,605   80.3
     FOV_013   72  54,197   69.8
     FOV_031   83  87,881   77.7
     FOV_015   95  52,204   72.4
     FOV_032  114  55,973   71.9
     FOV_019  246  72,619   73.1
  Spans 53 -> 246 cells (full density range). 401,479 total val spots.

scripts/segmenters.py
  Factory registry used by local_eval.py. Each factory takes (gpu, diameter)
  and returns a closure segment(fov_dir: Path) -> (2048, 2048) int mask. The
  model lives in the closure so it's built once and reused across FOVs.
  Available factories:
    build_all_background   — no-op, returns all-zeros. Plumbing smoke test.
    build_cellpose_zeroshot — wraps pipeline.py's segment_fov with cpsam.

scripts/local_eval.py
  Main validator. Arguments:
    --segmenter module.path:factory_name  (required)
    --val_fovs path  (default val_fovs.txt)
    --data_root path  (default /scratch/pl2820/data/competition)
    --gt_labels path  (default cache/gt_spot_labels.parquet)
    --gpu  (forwarded to the factory)
    --diameter float  (negative = auto; forwarded to the factory)
    --z_filter int  (NEW — diagnostic: restrict val spots to one z-plane
                     only. Filters both spots and GT in lockstep so the
                     length-match assertion still holds. Intended to
                     isolate per-z segmentation lift from over-/under-
                     segmentation errors.)
    --save_masks  (dumps predicted masks for debugging)
    --run_name name  (override runs/<name> dir)
  Workflow:
    1. Parse factory spec, build segmenter.
    2. Load cached GT for val FOVs only.
    3. For each val FOV: run segmenter, lookup spots in mask, assign
       '{fov}_cell_N' or 'background'.
    4. Log per-FOV: n_spots, n_pred_cells, n_gt_cells, pred_bg%, gt_bg%,
       ARI, seg_seconds.
    5. Build solution + submission DataFrames and call the organizer's
       metric.merfish_score VERBATIM (imported from
       /scratch/pl2820/data/competition/metric.py). This is the headline
       number. A parallel sklearn-based per-FOV ARI is logged for
       diagnostics; if the two disagree by > 1e-6, a warning is emitted.
  Run outputs in runs/<timestamp_or_name>/:
    config.json  — args + cuda_available + start time
    eval.log     — full log
    per_fov.csv  — one row per FOV with the stats above
    summary.json — mean_ari (official), mean_ari_inline, per_fov_ari
    masks/<fov>.npy  — only if --save_masks

run_local_eval.sh

scripts/compute_diameter.py                                        (Phase 2)
  Calibrates Cellpose's `diameter` hint from GT polygons. Parses
  cell_boundaries_train.csv, computes per-cell equivalent-circle diameter
  d = 2*sqrt(area_px/pi) via the shoelace formula on the stage-µm polygon
  (area_px = area_µm² / pixel_size² — the affine transform is isotropic so
  no per-FOV offsets are needed). Defaults to z=2 (the plane pipeline.py
  currently segments at); --all_z prints per-z stats; --z overrides. Writes
  the median to reference/diameter_px.txt.
  Result (z=2, 3774 / 4082 cells with valid polygon, 308 missing):
    mean=86.38 median=87.68 p25=74.76 p75=98.70 p90=107.79 (all in pixels)
  Cross-z distributions are tight (median per z: 90.6 / 87.4 / 87.7 / 89.6 /
  92.7), so z=2 is a fair single estimate.

scripts/train_cellpose.py                            (Phase 4, updated)
  Fine-tunes cpsam on the Phase 3 (multi-z) .npz files.

  File discovery: globs training_data/FOV_*_z*.npz, parses each stem with
  FOV_Z_RE = ^(FOV_[^_]+)_z(\d+)$, and groups files by FOV id — so the
  val split operates on whole FOVs (all 5 z-planes together) to prevent
  cross-z leakage between train and val at the image level.

  Deterministic Phase-4 val pick: ranks FOVs by z=2 mask cell count
  (canonical reference plane; falls back to max-across-z if a FOV has no
  z=2 file), takes inner-1/3 and inner-2/3 positions as the val set
  (count=2 by default). With 34 training FOVs, that leaves 32 FOVs in
  train (160 .npz files) and 2 FOVs in val (10 .npz files). The Phase 1
  val FOVs in val_fovs.txt are NOT used here — they are reserved for
  final ARI eval on this run's best weights.

  Calls cellpose.train.train_seg with the defaults: n_epochs=200,
  lr=1e-5, weight_decay=0.1, batch_size=8, save_every=20, save_each=True,
  normalize=False, channel_axis=-1. SGD flag is exposed for parity with
  the Phase 4 spec but cellpose v4.0.1+ always uses AdamW.

  Checkpointing (Option A — public API only, no per-epoch hooks):
    After train_seg returns, the script copies/reorganizes the raw
    cellpose saves into runs/<name>/checkpoints/:
      final.pt               — last-epoch weights
      epoch_NNNN.pt          — one per save_every (weights only)
      best.pt                — argmin(val_loss) over saved epochs
      best_meta.json         — {epoch, train_loss, val_loss, source, path}
    CAVEATS:
      - WEIGHTS ONLY. No optimizer state → no mid-run resume. A crashed
        or killed job loses everything after the last epoch_NNNN.pt.
      - val_loss is computed by cellpose only at epoch 5 and every 10
        epochs thereafter; with save_every=20 the candidate set for
        best.pt is exactly {20, 40, 60, 80, 100, 120, 140, 160, 180}.
      - Fallback: if no saved epoch has a populated val_loss (e.g. at
        n_epochs < 6), best.pt = final.pt with a noted "no eval epochs"
        message in best_meta.json.

  Outputs per run (runs/<name>/):
    config.json        — all hyperparams + val_fovs + train/val image counts + started_at
    train.log          — full training log (cellpose train_logger + this script)
    train_log.csv      — [epoch, train_loss, val_loss, lr] per epoch;
                         val_loss blank at non-eval rows; lr reconstructed
                         via compute_lr_schedule() (pinned to cellpose 4.1.1
                         semantics — warmup 10 ep, flat middle, step-halving
                         past epoch 150 at n_epochs=200)
    models/<name>      — cellpose's raw final-epoch checkpoint (source of final.pt)
    models/<name>_epoch_NNNN — cellpose's raw intermediate checkpoints (sources of epoch_NNNN.pt)
    checkpoints/       — normalized layout described above
    summary.json       — best_epoch, best_loss, final losses, val_fovs

Phase 5 scaffolding — per-spot z-plane lookup at inference time
---------------------------------------------------------------
Motivation: pipeline.py (and build_cellpose_finetuned) segments only z=2
and looks up ALL spots in that single 2D mask. Spots at z ∈ {0,1,3,4}
get assigned using a mask from the wrong z-plane, capping ARI for
those spots. The Phase 1 z-only diagnostic (phase1_zcheck_z2only:
0.4765 → 0.4932) already showed there's lift available from aligning
spot z with segmentation z — the signal was strongest on the densest
FOV (FOV_019 +0.12). Phase 5 closes that gap at inference time.

Approach: segment all 5 z-planes, stitch 2D masks across z via
Cellpose's stitch_threshold (IoU-based linking; same cell -> same id
across z's it appears in), then look up each spot in mask[global_z].
stitch_threshold=0.3 is a reasonable default per Cellpose 1.x-4.x
conventions; exposed via --segmenter_kwargs for tuning.

Files changed (scaffolding only — no training, no re-finetune):

scripts/segmenters.py  (+ build_cellpose_finetuned_3d)
  New factory. Signature matches the other factories plus
  stitch_threshold=0.3. On build:
    - Validates pretrained_model was passed.
    - Runs inspect.signature(models.CellposeModel.eval) and raises
      RuntimeError if 'stitch_threshold' is NOT in the parameter list.
      This is the Phase 5 spec's up-front compatibility check: if the
      installed cellpose can't stitch, we stop here rather than silently
      producing unstitched masks. Error message points to the Option A
      fallback (independent per-z segmentation — would be a separate
      factory). cellpose 4.1.1 definitely has it; the check protects
      against future rebuilds.
    - Loads cpsam-compatible weights from pretrained_model.
    - Precomputes POLY_T_FRAMES=[5,10,15,20,25], DAPI_FRAMES=[6,11,16,21,26]
      (Phase 3 verified frame map: polyT[z]=5+5z, DAPI[z]=6+5z).
  Per-FOV closure:
    - For z in 0..4: load_dax_frame(polyT), load_dax_frame(DAPI),
      normalize_image each (reuses pipeline.py helpers), build (H,W,3)
      = [polyT, DAPI, zeros].
    - Stack to (Z=5, H, W, 3) float32 and call
        model.eval(stack, diameter=..., channel_axis=-1, z_axis=0,
                   do_3D=False, stitch_threshold=0.3, normalize=False)
    - Returns (5, 2048, 2048) int32 mask. Shape guard raises if
      cellpose returns anything else.
  NOT wrapped in a list — passing the 4D array directly triggers
  cellpose's z-stack code path. Wrapping in [...] would iterate slices
  and skip stitching.

scripts/pipeline_v2.py
  Per-FOV:
    - Loads polyT (frames 5,10,15,20,25) and DAPI (frames 6,11,16,21,26)
      via pipeline.load_dax_frame; normalizes each per-z with
      pipeline.normalize_image; stacks to (Z=5, H, W, 3) float32.
    - Calls model.eval(stack, diameter=..., channel_axis=-1, z_axis=0,
      do_3D=False, stitch_threshold=0.3, normalize=False) — the same
      kwargs build_cellpose_finetuned_3d used in Phase 5 local eval.
    - Returns a (5, 2048, 2048) int32 stitched mask; shape-guards raise
      on anything else.
  Per-spot assignment:
    - Vectorized mask_stack[int(global_z), image_row, image_col] with
      clamps on all three axes (defense only; test schema is already in
      range: global_z ∈ {0,1,2,3,4} as float, row/col ∈ [0, 2047]).
    - Integer cell IDs → "cell_N"; 0 → "background".
  Up-front compatibility check: if inspect.signature of
  models.CellposeModel.eval doesn't expose stitch_threshold we raise
  before touching any FOV — same fail-fast policy as the segmenter
  factory, so we never silently produce unstitched masks.
  Post-write verification (identical to infer_test.py plus a per-FOV
  summary table):
    * columns == ['spot_id', 'fov', 'cluster_id']
    * row count == len(test_spots.csv) (= 224,495)
    * zero null cluster_ids
    * fov values ⊆ {FOV_A, FOV_B, FOV_C, FOV_D}
    * spot_id ordering matches sample_submission.csv row-for-row
    * logs per-FOV n_spots / n_cells / background% / seg_seconds —
      sanity signal before upload (expect 65–80% bg and ~50–300 cells
      per FOV, matching the val-FOV stats from runs/phase5_v1_eval)
  CLI: --pretrained_model --output --stitch_threshold (default 0.3)
       --diameter (None/negative = auto) --gpu --fovs (default TEST_FOVS)
       --data_root --spots --sample_submission.
